# SOA — Service-Oriented Architecture

**Суть:**
- Архитектура, где система строится на сервисах, которые взаимодействуют через стандартизированные интерфейсы (REST, SOAP, брокеры сообщений)
- Каждый сервис - независимый модуль, отвечающий за конкретную бизнес функцию

**Когда использовать:**
- Большие предприятия
- Когда нужно интегрировать много систем
- Часто с ESB (Enterprise Service Bus — Сервисная шина предприятия) и сложными корпоративными процессами

# MSA — Microservices Architecture

**Суть:**
- Развитие идеи SOA, сервисы мелкие и независимые, с отдельным жизненным циклом
- Каждый сервис:
	- Автономен
	- Имеет свою бд (или схему)
	- Общается через REST/gRPC/Events

**Когда использовать:**
- High-scale, быстро меняющиеся приложения
- Требуется независимая разработка команд

# Clean Architecture

**Суть:**
- Архитектурный стиль Роберта Мартина (Uncle Bob)
- Делит приложение на **слои**, чтобы **бизнес-логика не зависела от инфраструктуры**

**Слои (снаружи внутрь):**
```
Interface / UI Layer  
Infrastructure Layer (DB, Frameworks)  
Application Layer (Use Cases)  
Domain Layer (Entities, бизнес-логика)
```

**Правила:**
- Dependency Rule: зависимости идут внутрь, внутренняя логика не зависит от внешнего мира
- Тестируемость: Domain и Use Cases легко тестировать отдельно
- Инфраструктура меняется, бизнес — нет

**Связь с DI:**
- DI-контейнер внедряет внешние зависимости (репозитории, API-клиенты) в слой Use Cases/Services

# DDD — Domain-Driven Design

**Суть:**
- Подход к проектированию ПО вокруг доменной модели
- Основная цель: сделать код отражением реальной бизнес-логики

## Основные концепции DDD:

|Концепция|Что это|
|---|---|
|Entity|объект с уникальной идентичностью|
|Value Object|объект без идентичности, только данные|
|Aggregate|группа связанных объектов, которые управляются как единое целое|
|Repository|интерфейс для доступа к Aggregate|
|Domain Service|операции, которые не принадлежат конкретной сущности|
|Bounded Context|граница модели, внутри которой правила домена согласованы|

**Связь с Clean Architecture:**
- DDD модели лежат в Domain Layer
- Use Cases работают с агрегатами и репозиториями

**Связь с MSA:**
- Часто один микросервис реализует один Bounded Context
- MSA помогает изолировать контексты и команды

# Как они связаны

```
MSA     → deployment и масштабирование  
SOA     → крупные интегрируемые сервисы  
Clean   → внутренняя архитектура приложения  
DDD     → организация бизнес-логики и модели
```

**Пример:**
- Микросервис UserService (MSA)
	- Внутри Clean Architecture: UI → Use Cases → Domain → Repository
	- Domain строится по DDD: User Entity, User Aggregate, UserRepository

---

💡 **Итоговые рекомендации:**
- SOA   → для крупных интегрируемых систем
- MSA   → для agile, масштабируемых приложений
- Clean → для тестируемой и поддерживаемой структуры кода
- DDD   → для сложной бизнес-логики, чтобы код отражал бизнес
